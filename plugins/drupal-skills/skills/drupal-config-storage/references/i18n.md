# Configuration Translation Patterns

This reference covers how to make Drupal configuration translatable. Configuration translation is separate from interface translation (`t()` / `TranslatableMarkup`) -- it translates stored config values like site name, field labels, and module settings.

## How config translation works

1. Config schema defines which fields are translatable (via `type: label` or `type: text` or explicit `translatable: true`)
2. When config is installed from `config/install/`, Drupal registers translatable strings in the translation system
3. Admins translate config values at `/admin/config/regional/config-translation`
4. At runtime, Drupal applies language overrides based on the current language context

## Making config fields translatable

### Use translatable schema types

The `label` and `text` types are translatable by default. The `string` type is NOT translatable.

```yaml
# config/schema/my_module.schema.yml
my_module.settings:
  type: config_object
  label: 'My Module settings'
  mapping:
    site_greeting:
      type: label
      label: 'Site greeting'
    help_text:
      type: text
      label: 'Help text displayed to users'
    api_key:
      type: string
      label: 'API key'
```

In this example, `site_greeting` (label) and `help_text` (text) are translatable. `api_key` (string) is not -- which is correct because API keys should not vary by language.

### Explicit translatable flag

You can explicitly mark any type as translatable or non-translatable:

```yaml
my_module.settings:
  type: config_object
  label: 'My Module settings'
  mapping:
    custom_message:
      type: string
      label: 'Custom message'
      translatable: true
    internal_identifier:
      type: label
      label: 'Internal ID'
      translatable: false
```

## Interface translation vs config translation

| Aspect | Interface translation | Config translation |
|--------|----------------------|-------------------|
| What it translates | Code strings (`$this->t('Hello')`) | Stored config values (site name, field labels) |
| Where strings come from | PHP code, Twig templates | `config/install/*.yml` files, admin UI settings |
| How to translate | `/admin/config/regional/translate` or `.po` files | `/admin/config/regional/config-translation` |
| Storage | `locales_source` / `locales_target` tables | Config override system (language collection) |
| Requires | `locale` module | `config_translation` module |

## Config translation override system

Config translations are stored as language-specific config overrides. When Drupal loads a config object in a French context, it applies the French override automatically.

### Reading translated config

In normal operation, `\Drupal::config()` and `$this->config()` automatically return the translated value for the current language. No special code is needed.

### Reading config in a specific language

```php
$language_manager = \Drupal::service('language_manager');
$french = $language_manager->getLanguage('fr');
$original = $language_manager->getConfigOverrideLanguage();

// Temporarily switch to French:
$language_manager->setConfigOverrideLanguage($french);
$config = \Drupal::config('my_module.settings');
$greeting = $config->get('site_greeting'); // French value

// Restore original language:
$language_manager->setConfigOverrideLanguage($original);
```

### Editing config ignores overrides

`$factory->getEditable()` always returns the original (untranslated) stored value. This prevents admins from accidentally saving a translated override as the base config value.

## Making a module's settings translatable -- complete example

### 1. Config install file

```yaml
# config/install/weather_widget.settings.yml
location_label: 'Current Weather'
description: 'Shows the current weather for your area.'
api_key: ''
```

### 2. Config schema with translatable types

```yaml
# config/schema/weather_widget.schema.yml
weather_widget.settings:
  type: config_object
  label: 'Weather Widget settings'
  mapping:
    location_label:
      type: label
      label: 'Location label'
    description:
      type: text
      label: 'Widget description'
    api_key:
      type: string
      label: 'API key'
```

`location_label` (type: label) and `description` (type: text) are automatically translatable. `api_key` (type: string) is not.

### 3. Enable translation

With the `config_translation` module enabled, translations for this config appear at:
`/admin/config/regional/config-translation`

No additional code is needed. The schema tells Drupal which fields can be translated.

## Key rules

- Use `type: label` for short translatable text (names, titles, button labels)
- Use `type: text` for longer translatable text (descriptions, help text, messages)
- Use `type: string` for non-translatable values (API keys, machine names, URLs)
- The `config_translation` module must be enabled for config translation UI
- Config installed from `config/install/` is automatically registered with the translation system
- `getEditable()` always returns the untranslated base value -- this is by design
