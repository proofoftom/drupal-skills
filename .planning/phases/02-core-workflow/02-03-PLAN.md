**SKILL.md body (under 500 lines):**

1. **Decision tree: "What kind of plugin do you need?"**
   - Custom block (content block placed via Block Layout)? -> Extend `BlockBase`, implement `build()`. Add DI via `ContainerFactoryPluginInterface`.
   - Block with configuration form? -> Same as above + implement `blockForm()`, `blockSubmit()`, `blockValidate()`.
   - Custom plugin type (reusable extension point)? -> Create plugin manager + annotation/attribute class + plugin interface.
   - Existing plugin type (Field formatter, Field widget, etc.)? -> Extend the appropriate base class with correct annotation/attribute.

2. **Block plugin -- complete pattern**
   - Class extends `BlockBase`
   - D10 annotation: `@Block(id = "machine_name", admin_label = @Translation("Human Label"))`
   - D11 attribute: `#[Block(id: "machine_name", admin_label: new TranslatableMarkup("Human Label"))]`
   - `build()` method returns render array
   - Show BOTH D10 and D11 syntax side by side with import differences noted
   - Namespace: `Drupal\module_name\Plugin\Block`
   - File location: `src/Plugin/Block/MyBlock.php`

3. **Plugin DI -- THE critical pattern**
   - Implement `ContainerFactoryPluginInterface`
   - `create()` takes 4 params: `$container, $configuration, $plugin_id, $plugin_definition` -- NOT just `$container` like controllers
   - Constructor takes `$configuration, $plugin_id, $plugin_definition` PLUS injected services
   - Constructor MUST call `parent::__construct($configuration, $plugin_id, $plugin_definition)`
   - WRONG-WAY CALLOUT: Using controller DI signature in plugins. Plugin `create()` has 4 parameters, not 1. Missing `$configuration, $plugin_id, $plugin_definition` causes fatal errors.
   - WRONG-WAY CALLOUT: Forgetting `parent::__construct($configuration, $plugin_id, $plugin_definition)` in plugin constructor. Without this, plugin configuration (including block placement settings) is lost.
   - WRONG-WAY CALLOUT: Using `\Drupal::service()` in plugin classes -- inject via `create()` + constructor, same rule as controllers and forms.
   - Show complete block with DI example (both D10 and D11 syntax)

4. **Block configuration forms**
   - `blockForm($form, FormStateInterface $form_state)`: add form elements to block config
   - `blockSubmit($form, FormStateInterface $form_state)`: save config via `$this->setConfigurationValue('key', $value)`
   - `blockValidate($form, FormStateInterface $form_state)`: optional validation
   - Access stored config: `$this->configuration['key']`
   - `defaultConfiguration()`: provide default values for config keys
   - Block config is stored automatically by the block system -- do NOT manually save to Config API
   - WRONG-WAY CALLOUT: Manually saving block config to Config API. Block config is managed by the block placement system. Use `$this->setConfigurationValue()` in `blockSubmit()` and `$this->configuration` to read.

5. **Block access control**
   - Override `blockAccess(AccountInterface $account)` to control who sees the block
   - Return `AccessResult::allowed()`, `::forbidden()`, or `::neutral()`
   - Must include cache contexts: `->addCacheContexts(['user.permissions'])`
   - Cross-reference drupal-access-security for full access system

6. **Custom plugin types**
   - Three components needed: (1) Plugin Manager, (2) Annotation/Attribute class, (3) Plugin Interface
   - **Plugin Manager:** extends `DefaultPluginManager`, registered as service
     - Constructor: `parent::__construct('Plugin/PluginType', $namespaces, $module_handler, PluginInterface::class, AnnotationClass::class)`
     - D11: use `AttributeClassDiscovery` or `AttributeBridgeDecorator` for backward compat
   - **Annotation class (D10):** in `src/Annotation/`, extends `\Drupal\Component\Annotation\Plugin`
   - **Attribute class (D11):** extends `\Drupal\Component\Plugin\Attribute\Plugin`
   - **Plugin interface:** extends `PluginInspectionInterface` (and often `ContainerFactoryPluginInterface`)
   - **Plugin base class:** optional abstract class implementing the interface
   - Show complete custom plugin type example with all three components + services.yml
   - WRONG-WAY CALLOUT: Using `@Translation()` in D11 attribute syntax. D11 attributes use `new TranslatableMarkup()`. @Translation is annotation-only syntax.

7. **Plugin discovery**
   - Annotation-based (D10): Doctrine annotation reader scans `src/Plugin/PluginType/` namespace
   - Attribute-based (D11): PHP native attribute reader scans same namespace
   - YAML-based: some plugin types use YAML discovery (e.g., menu links) -- different system
   - Plugin directory convention: `src/Plugin/{PluginType}/`

8. **Cross-references**
   - "See also: drupal-forms-api (if installed) for Form API lifecycle used in block config forms (blockForm/blockSubmit). If not available, blockForm() uses the same render array form elements as buildForm()."
   - "See also: drupal-config-storage (if installed) for Config API patterns. Block config is NOT stored via Config API -- it uses the block placement system. Config API is for module-level settings."
   - "See also: drupal-access-security (if installed) for the full access control system including AccessResult cache metadata. If not available, use AccessResult::allowedIfHasPermission() which handles caching automatically."
   - "See also: drupal-routing-controllers (if installed) for service registration patterns (.services.yml). Plugin managers are registered as services."
   - "See also: drupal-entities-fields (if installed) for entity-level plugins (field formatters, field widgets, entity handlers)."
