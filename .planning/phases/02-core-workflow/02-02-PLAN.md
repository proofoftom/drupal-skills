**SKILL.md body (under 500 lines):**

1. **Decision tree: "Where should you store this data?"**
   - Admin-configurable settings (site name, API keys, feature toggles)? -> **Config API**. Stored in YAML, exportable between environments, requires schema.
   - System state (last cron run, flag, environment marker)? -> **State API**. Key/value store, NOT exportable, environment-specific.
   - Temporary per-user data (form drafts, wizard progress, locks)? -> **TempStore**. Auto-expires. PrivateTempStore for per-user, SharedTempStore for multi-user coordination.
   - Structured data with UI (products, categories, workflows)? -> Not storage API -- use a config entity (see drupal-entities-fields skill).

2. **Config API**
   - Reading config: `\Drupal::config('module.settings')->get('key')` (immutable) vs `\Drupal::configFactory()->getEditable('module.settings')` (mutable, for saving)
   - In classes: inject `config.factory` service, use `$this->configFactory->get()` or `$this->configFactory->getEditable()`
   - Saving config: `$config->set('key', $value)->save()`
   - Default config values: `config/install/module_name.settings.yml`
   - Config schema: `config/schema/module_name.schema.yml` -- REQUIRED for every config object
   - WRONG-WAY CALLOUT: Missing config schema file -- config "works" without schema but breaks config export/import validation, translation, and type casting. Always create config/schema/module_name.schema.yml.
   - WRONG-WAY CALLOUT: Using `variable_get()`/`variable_set()` -- this is Drupal 7. Use Config API or State API.
   - Show complete example: config/install/module_name.settings.yml (defaults) + config/schema/module_name.schema.yml (schema) + reading in a service

3. **Config schema deep dive**
   - Schema file structure: `module_name.settings:` with `type: config_object`, `label:`, `mapping:` containing typed keys
   - Common types: `string`, `text`, `integer`, `boolean`, `float`, `email`, `uri`, `label` (translatable string), `path`
   - Nested mappings and sequences
   - `config_object` vs `config_entity` type
   - WRONG-WAY CALLOUT: Using `type: string` for translatable text -- use `type: label` to enable config translation. Plain `string` is never translated.

4. **State API**
   - Reading: `\Drupal::state()->get('module_name.key', $default)`
   - Writing: `\Drupal::state()->set('module_name.key', $value)`
   - In classes: inject `state` service (`Drupal\Core\State\StateInterface`)
   - Multiple operations: `getMultiple()`, `setMultiple()`, `delete()`, `deleteMultiple()`
   - No schema needed. Not exportable. Cleared on cache rebuild if desired.
   - WRONG-WAY CALLOUT: Storing admin settings in State API -- State is NOT exportable. Settings configured by admins belong in Config API so they transfer between environments. State is for runtime flags like "last cron timestamp."
   - Show service example with injected StateInterface

5. **TempStore**
   - PrivateTempStore: per-user temporary storage (form wizard data, multistep progress)
   - SharedTempStore: multi-user coordination (content locking, shared editing state)
   - Inject `tempstore.private` or `tempstore.shared` service
   - Usage: `$store = $this->tempStoreFactory->get('module_name'); $store->set('key', $data); $data = $store->get('key'); $store->delete('key');`
   - Auto-expires (configurable, default 604800 seconds / 7 days)
   - Brief complete example

6. **Config overrides**
   - Environment-specific: `$config['module.settings']['key'] = 'value';` in settings.php
   - Language overrides: handled by config translation system (see references/i18n.md)
   - Module overrides: `ConfigFactoryOverrideInterface`
   - Note: overrides are read-only in `$this->config()` -- `getEditable()` bypasses overrides

7. **Cross-references**
   - "See also: drupal-forms-api (if installed) for ConfigFormBase which provides a form UI for Config API settings. If not available, use ConfigFormBase extending FormBase with getEditableConfigNames()."
   - "See also: drupal-entities-fields (if installed) for config entities when you need structured, CRUD-able configuration with admin UI. If not available, config entities are PHP classes with ConfigEntityType annotation."
   - "See also: drupal-module-scaffold (if installed) for .info.yml setup and config/install directory. If not available, place default config in config/install/module_name.settings.yml."
   - "For configuration translation patterns, see references/i18n.md in this skill directory."
