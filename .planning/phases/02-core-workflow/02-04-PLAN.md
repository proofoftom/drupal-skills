**SKILL.md body (under 500 lines):**

1. **Decision tree: "What kind of access control do you need?"**
   - Route requires a specific permission? -> `_permission: 'permission name'` in .routing.yml + define in permissions.yml
   - Route requires a specific role? -> `_role: 'role_name'` (less common, prefer permissions)
   - Route needs custom logic (ownership, status, conditions)? -> `_custom_access: '\Drupal\module\Access\Checker::method'`
   - Route needs CSRF protection (state-changing action links)? -> `_csrf_token: 'TRUE'` in requirements
   - Entity needs access control? -> Implement `EntityAccessControlHandler`
   - Block needs access control? -> Override `blockAccess()` on block plugin
   - Open access (rare, public pages)? -> `_access: 'TRUE'` -- use only for truly public routes

2. **Permissions**
   - Define in `module_name.permissions.yml`:
     ```yaml
     administer module_name:
       title: 'Administer Module Name'
       description: 'Full admin access to Module Name.'
       restrict access: true
     access module_name content:
       title: 'Access Module Name content'
     ```
   - `restrict access: true`: shows warning on permissions page (for dangerous permissions)
   - Dynamic permissions via `permission_callbacks` in permissions.yml
   - WRONG-WAY CALLOUT: Referencing a permission string in route `_permission` that isn't defined in permissions.yml. Every permission string used in routes or access checks MUST be defined in module_name.permissions.yml. Orphaned permission strings silently deny access.
   - WRONG-WAY CALLOUT: Using `hook_permission()` -- this is Drupal 7. Define permissions in YAML file.

3. **Route access patterns**
   - `_permission: 'permission name'`: single permission check (most common)
   - `_permission: 'perm1+perm2'`: require ALL permissions (AND)
   - `_permission: 'perm1,perm2'`: require ANY permission (OR)
   - `_role: 'administrator'`: role check (prefer permissions over roles)
   - `_custom_access: '\Drupal\module\Access\MyChecker::access'`: custom logic
   - `_access: 'TRUE'`: open access (truly public pages only)
   - Show complete .routing.yml examples for each pattern

4. **Custom access checkers**
   - Class method returns `AccessResult` (allowed, forbidden, neutral)
   - Method signature: `public function access(AccountInterface $account, ...)` -- additional params auto-resolved from route parameters
   - `AccessResult::allowed()`: grant access
   - `AccessResult::forbidden('reason')`: deny access (with optional reason for logging)
   - `AccessResult::neutral()`: no opinion (other checks may still grant/deny)
   - `AccessResult::allowedIfHasPermission($account, 'permission')`: shortcut that handles caching automatically
   - `AccessResult::allowedIf($condition)`: conditional access
   - WRONG-WAY CALLOUT: Returning bare `AccessResult::allowed()` or `::forbidden()` without cache metadata. Access results are cached! Always add `->addCacheContexts(['user.permissions'])` or use `allowedIfHasPermission()` which handles this automatically. Without cache metadata, access decisions become stale.
   - Show complete custom access checker example with cache contexts and tags

5. **Entity access control**
   - Define `EntityAccessControlHandler` subclass
   - Set in entity annotation/attribute: `access = "Drupal\module\MyAccessHandler"`
   - Override `checkAccess()` for view/update/delete operations
   - Override `checkCreateAccess()` for create operation
   - Return `AccessResult` with proper cache metadata
   - Default: `EntityAccessControlHandler` checks `administer {entity_type}` permission
   - Brief example

6. **CSRF protection**
   - For state-changing links (not forms -- forms have built-in CSRF tokens):
     ```yaml
     module_name.action:
       path: '/module/action/{entity}'
       requirements:
         _csrf_token: 'TRUE'
     ```
   - Links built with `Url::fromRoute()` automatically get CSRF token appended
   - Drupal validates the token on request -- no manual token handling needed
   - WRONG-WAY CALLOUT: Manually generating and checking CSRF tokens. Use `_csrf_token: 'TRUE'` route requirement and build links with `Url::fromRoute()`. Drupal handles token generation and validation automatically.

7. **XSS prevention**
   - Twig auto-escapes by default -- raw output requires explicit `|raw` filter
   - In PHP render arrays: `#markup` is auto-filtered, `#plain_text` is always escaped
   - For user input in PHP: `Html::escape($string)` for plain text, `Xss::filter($html)` for filtered HTML, `Xss::filterAdmin($html)` for admin HTML
   - `$this->t()` / `new TranslatableMarkup()` auto-escapes placeholders with `@` prefix
   - Placeholder types: `@variable` (escaped), `%variable` (escaped + wrapped in em), `:url` (escaped + URL-validated)
   - WRONG-WAY CALLOUT: Using `#markup` with unsanitized user input. `#markup` allows a safe HTML subset but does NOT fully escape. For untrusted text, use `#plain_text` which escapes everything, or explicitly sanitize with `Html::escape()`.

8. **Cross-references**
   - "See also: drupal-routing-controllers (if installed) for route definitions where access requirements are declared. If not available, access is set in module_name.routing.yml under requirements."
   - "See also: drupal-entities-fields (if installed) for entity type definitions that reference access handlers. If not available, set access handler in entity annotation/attribute."
   - "See also: drupal-plugins-blocks (if installed) for block-level access via blockAccess(). If not available, override blockAccess() on BlockBase subclass returning AccessResult."
   - "See also: drupal-forms-api (if installed) for form-level security. Forms have built-in CSRF protection via form tokens -- CSRF route requirement is for non-form links."
