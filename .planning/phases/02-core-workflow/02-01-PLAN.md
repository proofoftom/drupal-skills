**SKILL.md body (under 500 lines):**

1. **Decision tree: "What kind of form do you need?"**
   - Configuration/settings form? -> Extend `ConfigFormBase`. Implement `getEditableConfigNames()`, `getFormId()`, `buildForm()`, `submitForm()`. Pair with config schema YAML.
   - Standalone form (contact, search, custom)? -> Extend `FormBase`. Implement `getFormId()`, `buildForm()`, `submitForm()`, optionally `validateForm()`.
   - Confirmation form (delete, irreversible action)? -> Extend `ConfirmFormBase`. Implement `getQuestion()`, `getCancelUrl()`, `submitForm()`.
   - Altering an existing form? -> Use `hook_form_alter()` or `hook_form_FORM_ID_alter()` in .module file.

2. **FormBase lifecycle**
   - `getFormId()`: unique machine name for the form
   - `buildForm()`: return render array with form elements (#type, #title, #default_value, #required)
   - `validateForm()`: use `$form_state->setErrorByName('field_name', $this->t('Error'))` to flag errors
   - `submitForm()`: process values via `$form_state->getValue('field_name')`, set redirect, add message
   - Show DI pattern: inject services via `create()` + constructor (same pattern as controllers)
   - WRONG-WAY CALLOUT: Using `drupal_set_message()` -- deprecated since D8.5. Use `$this->messenger()->addMessage()`.
   - WRONG-WAY CALLOUT: Validation with exceptions or drupal_set_message instead of `$form_state->setErrorByName()`. setErrorByName highlights the specific field and prevents submission.
   - WRONG-WAY CALLOUT: Using `\Drupal::service()` inside form classes -- inject via `create()` + constructor. Static calls only in .module procedural code.

3. **ConfigFormBase for settings forms**
   - Extends FormBase with config helpers: `$this->config('module.settings')` in buildForm, `$this->config('module.settings')->set()->save()` in submitForm
   - `getEditableConfigNames()`: returns array of config names the form edits
   - `parent::buildForm()` adds the submit button automatically
   - `parent::submitForm()` adds "saved" message automatically
   - MUST pair with `config/schema/module_name.schema.yml` -- config forms without schemas break export, validation, and translation
   - Show complete ConfigFormBase example with route, form class, and config schema (3 files)
   - WRONG-WAY CALLOUT: Missing config schema file -- form "works" in dev but breaks config export/import and translation. Always create config/schema/module_name.schema.yml.

4. **ConfirmFormBase for destructive actions**
   - `getQuestion()`: "Are you sure you want to delete X?"
   - `getCancelUrl()`: where to go on cancel
   - `getDescription()`: optional explanation
   - `getConfirmText()`: button label (defaults to "Confirm")
   - Brief complete example

5. **Form altering**
   - `hook_form_alter(&$form, FormStateInterface $form_state, $form_id)`: fires for ALL forms
   - `hook_form_FORM_ID_alter(&$form, FormStateInterface $form_state, $form_id)`: fires for ONE specific form
   - WRONG-WAY CALLOUT: Using `hook_form_alter()` without checking `$form_id` -- accidentally alters ALL forms. Use `hook_form_FORM_ID_alter()` when targeting a specific form, or add `if ($form_id === 'target_form')` guard.
   - Adding custom submit/validate handlers: `$form['#submit'][]` and `$form['#validate'][]`
   - Adding form elements to existing forms
   - Show example of altering user registration form

6. **Form elements quick reference** (brief table, not exhaustive -- per REQUIREMENTS.md out-of-scope)
   - Common types: textfield, textarea, select, checkboxes, radios, number, email, password, submit, hidden, value
   - Key properties: #type, #title, #default_value, #required, #description, #states (conditional visibility)
   - Note: full element reference at api.drupal.org/api/drupal/elements

7. **Route for forms**
   - Use `_form` default in .routing.yml (not `_controller`)
   - Show .routing.yml entry pointing to form class
   - Admin forms: add `_admin_route: TRUE` option

8. **Cross-references**
   - "See also: drupal-routing-controllers (if installed) for route definitions and service DI patterns. If not available, define routes in module_name.routing.yml with _form default."
   - "See also: drupal-config-storage (if installed) for Config API details, State API, and TempStore. If not available, use ConfigFormBase pattern shown above for config forms."
   - "See also: drupal-entities-fields (if installed) for entity form handlers and field widgets. If not available, entity forms use the same Form API lifecycle."
   - "See also: drupal-module-scaffold (if installed) for .module file setup where form alter hooks live."

Extract content from book Ch 2 (Form API sections: FormBase, ConfigFormBase, form altering, buildForm/validateForm/submitForm lifecycle, submit handlers).
