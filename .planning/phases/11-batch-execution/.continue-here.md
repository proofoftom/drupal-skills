<context>
The user wants eval verification that's meaningful, not theatrical. The guiding principle is "put your Behat hat on" — every E2E check should exercise an actual code path that would surface real errors, not just confirm something exists in the admin UI.

The eval pipeline flow for this milestone:
1. eval-executor creates the module (with-skill and without-skill variants)
2. eval-grader checks code expectations (static analysis of generated files)
3. eval-browser runs ONLY for skills with genuinely useful UI checks (access, forms, entities, routing, views, database reports)
4. phpunit runs for the testing skill specifically
5. phpcs runs as a static analysis expectation for all skills

The user is thinking ahead to a next milestone where evals build one integrated module (like a custom AI module for drupal.org) testing all skills together. That's where full browser-based E2E with real content and proper UAT would shine. For now, the focus is getting a baseline with static analysis + targeted E2E.
