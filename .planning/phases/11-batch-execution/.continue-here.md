<critical_insight>
## Runtime verification > code review (user directive, session 12)

Current grader does code review — checks if patterns exist in source files. This is
fundamentally flawed: "the code looks right" ≠ "the code works." Grader can pass
getCacheContexts() as present without knowing if the block actually renders or if
the cache contexts actually take effect.

**Expectations should test runtime behavior, not source patterns.**

Examples of runtime-verifiable expectations:
- "ddev drush php-eval loads the custom service from container and calls a method without fatal"
- "curl to the block placement page shows the block is discoverable in Block Layout"
- "eval-browser verifies the block renders content on a node page (not empty, not error)"
- "ddev drush php-eval creates a node, loads the block render array, and confirms #cache.contexts contains 'user'"
- "curl to /api/endpoint returns JSON with expected keys (not 403, not 500)"

Tools available:
- `ddev drush php-eval '...'` — execute PHP in Drupal bootstrap, test services/entities/render
- `curl -sk https://project.ddev.site/path` — HTTP-level E2E
- eval-browser agent — UI verification via agent-browser skill (drush uli for auth)

This naturally differentiates: sloppy code that "looks right" often fatals at runtime.
Code that actually works through Drupal's full stack (plugin discovery, DI, render, cache)
is harder to produce without proper patterns — exactly what the skills teach.

This may also reduce or eliminate the scaffold confound: if the without-skill model
produces code that looks right but doesn't actually work, the runtime tests will catch it
regardless of how "Drupal developer"-like the agent prompt makes it sound.
</critical_insight>
