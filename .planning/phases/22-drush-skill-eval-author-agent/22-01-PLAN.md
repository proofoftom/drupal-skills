**Required sections (CRITICAL patterns first, consistent with v2.0 learning: content placement drives delta):**

1. **drush generate — scaffold boilerplate, save tokens** (CRITICAL — first section)
   - What drush generate does and why agents should use it
   - Key generators: module, controller, form:config, form:simple, entity:content, entity:configuration, plugin:block, service-provider, event-subscriber, hook, test:kernel
   - Using `--answer` for non-interactive generation (key for agent automation)
   - Using `--dry-run` to preview output
   - Generates phpcs-compliant code by default
   - WRONG/RIGHT: manually writing 200 lines of content entity boilerplate vs `drush generate entity:content`
   - Note: In ddev environments, prefix with `ddev drush generate`

2. **Self-verification recipes — check your own work** (CRITICAL — second section)
   This is the core value. Agents should verify after every major operation.

   Recipe: After creating/modifying routes:
   ```bash
   drush route --name=my_module.my_route        # verify route exists
   drush route --path=/admin/my-module/items     # verify path maps correctly
   ```
   WRONG/RIGHT: complex multi-line php-eval with router.route_provider vs one-line `drush route`

   Recipe: After any module change (ALWAYS do this):
   ```bash
   drush cr                                       # rebuild caches
   drush watchdog:show --severity-min=Warning --count=5   # check for new errors
   ```

   Recipe: Service/DI verification:
   ```bash
   drush php-eval "echo \Drupal::hasService('my_module.service') ? 'OK' : 'MISSING';"
   drush php-eval "\$s = \Drupal::service('my_module.service'); echo get_class(\$s);"
   ```

   Recipe: Permission verification:
   ```bash
   drush role:list --format=json
   drush php-eval "echo count(\Drupal::service('user.permissions')->getPermissions());"
   ```

   Recipe: Config/state inspection:
   ```bash
   drush config:get my_module.settings
   drush state:get my_module.last_run
   ```

   Recipe: Queue verification:
   ```bash
   drush queue:list                              # see queues and item counts
   drush queue:run my_module_queue               # process items
   ```

3. **The Drupal-first principle — entity API over SQL** (CRITICAL — third section)
   - WRONG/RIGHT: `drush sql:query "UPDATE node SET status=1 WHERE nid=42"` vs `drush entity:save node 42 --publish`
   - sql:query bypasses Drupal's entity API: hooks, cache invalidation, access checks ALL skipped
   - entity:create for creating test entities (interactive, fires creation hooks)
   - entity:save for re-saving (fires hook_entity_presave, hook_entity_update)
   - entity:delete for cleanup (fires hook_entity_delete, cleans references)
   - php-eval with entity API for programmatic testing of custom entities:
   ```bash
   drush php-eval "\$e = \Drupal::entityTypeManager()->getStorage('task')->create(['title' => 'Test']); \$e->save(); echo \$e->id();"
   ```
   - Use sql:query ONLY for raw DB inspection with no API equivalent (e.g., checking raw table structure, verifying schema)
   - WRONG/RIGHT: using sql:query to inspect entity field values vs loading the entity with php-eval and reading field values through the API

4. **Debugging with Drush**
   - watchdog:show patterns:
     - `--type=php` for PHP errors
     - `--severity-min=Warning` for all warnings and above
     - `--type=my_module` for module-specific logs
     - `--format=json` for machine-readable output
   - watchdog:tail for live monitoring during development
   - core:status for environment checks
   - core:route for route debugging (also covered in self-verification)
   - config:get / config:export for config debugging
   - state:get / state:set for state debugging
   - cache:tags for testing cache tag invalidation

5. **php:eval vs php:script — when to use each**
   - php:eval for one-liners: service resolution, class existence, simple checks
   - php:script for complex multi-step tests: entity workflows, DI verification, integration scenarios
   - Shell escaping tips for php:eval (single quotes outside, double inside, or heredoc)
   - WRONG/RIGHT: 10-line shell-escaped php-eval vs a php:script file
   - Example php:script for entity workflow testing

6. **Cross-references**
   - drupal-module-scaffold: drush generate complements manual scaffolding (generate the skeleton, customize the details)
   - drupal-batch-queue-cron: queue:list/queue:run for testing queue workers
   - drupal-testing: drush test:run for PHPUnit execution
   - For creating custom Drush commands: see references/command-authoring.md

**Style rules:**
- Use `>` blockquote format for WRONG/RIGHT callouts
- Place CRITICAL patterns in the first 3 sections
- Include code examples with inline comments
- Commands shown without `ddev` prefix (note ddev prefix in intro)
- Do NOT duplicate content from cross-referenced skills
