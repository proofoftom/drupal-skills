**SKILL.md body (under 500 lines):**

1. **Decision tree: "What are you rendering?"**
   - Simple text/markup? -> Use `#markup` (sanitized by Xss::filterAdmin)
   - Plain text (fully escaped)? -> Use `#plain_text`
   - Custom themed output? -> Define `hook_theme()`, use `#theme`, create Twig template
   - Existing core theme hook (table, item_list, links)? -> Use `#theme` with core hook name
   - Render element (standardized component)? -> Use `#type` (form elements, containers, etc.)

2. **Render arrays -- the core concept**
   - Every controller/block/form returns a render array
   - Keys starting with # are properties (#theme, #markup, #cache, #attached, #prefix, #suffix)
   - Keys NOT starting with # are children (nested render arrays)
   - Render arrays are NOT HTML -- they are declarative descriptions that Drupal renders
   - WRONG-WAY CALLOUT: Returning raw HTML strings from controllers. Always return render arrays. Strings bypass caching, altering, and security filtering.

3. **Defining a theme hook (hook_theme)**
   - Implement `hook_theme()` in .module file
   - Returns array of theme hook definitions with 'variables' key
   - Variables define the data contract between PHP and Twig
   - Always provide default values for all variables (NULL or empty)
   - WRONG-WAY CALLOUT: Omitting variables in hook_theme(). Every variable the template uses must be declared with a default value, or it will be undefined in Twig.
   - Show complete hook_theme() example from book (hello_world_salutation)

4. **Template naming and location**
   - Hook name `module_name_feature` maps to template `module-name-feature.html.twig`
   - Rule: underscores become hyphens in template filename
   - Templates go in `templates/` directory inside module
   - WRONG-WAY CALLOUT: Template name not matching hook name (forgetting underscore-to-hyphen conversion). Template `my_module_block` must be `my-module-block.html.twig`.

5. **Twig template patterns**
   - `{{ variable }}` for output (auto-escaped)
   - `{% if condition %}` for logic
   - `{{ attributes }}` for the attributes object (classes, id, data-*)
   - Filters: `|t` for translation, `|raw` (avoid -- security risk), `|escape`
   - Twig auto-escapes output -- this is by design, do NOT use `|raw` unless you know content is safe
   - Show template example from book with attributes and conditionals

6. **Preprocess functions**
   - `template_preprocess_HOOK(&$variables)`: default preprocessor (in .module file)
   - `MODULE_preprocess_HOOK(&$variables)`: module-specific preprocessor
   - Use to add computed variables, set default attributes, prepare data for Twig
   - `$variables['attributes']` is an Attribute object with addClass(), setAttribute() methods
   - Show preprocess example adding CSS classes

7. **Libraries API (CSS/JS attachment)**
   - Define libraries in `module_name.libraries.yml`
   - Attach via `$render['#attached']['library'][] = 'module_name/library_name'`
   - Library YAML: version, css (with weight categories: base/layout/component/state/theme), js, dependencies
   - Dependencies: list core libraries needed (core/jquery, core/drupal, core/once)
   - WRONG-WAY CALLOUT: Using inline `<script>` or `<style>` tags. Always use libraries API. Inline tags bypass aggregation, dependency management, and conditional loading.
   - Show complete libraries.yml example

8. **Common core theme hooks**
   - `#theme => 'table'`: #header (array of strings), #rows (array of row arrays), #empty (message when no rows)
   - `#theme => 'item_list'`: #items (array), #list_type ('ul' or 'ol'), #title
   - `#theme => 'links'`: #links (array of title/url pairs), #set_active_class
   - Show examples of each

9. **Attaching drupalSettings (PHP to JS)**
   - `$render['#attached']['drupalSettings']['module_name']['key'] = $value`
   - Access in JS: `settings.module_name.key`
   - Values must be scalar or simple arrays (JSON-serializable)

10. **Cross-references**

**references/js-ajax.md (no line limit, aim for ~200-300 lines):**

Cover from book Ch 12:

1. **Drupal.behaviors pattern** -- the correct way to attach JS
   - `Drupal.behaviors.behaviorName = { attach: function(context, settings) { ... } }`
   - `context` is the DOM element being processed (document on initial load, Ajax-inserted element on Ajax)
   - `settings` is drupalSettings object
   - Behaviors re-fire on every Ajax/BigPipe load -- this is by design
   - WRONG-WAY CALLOUT: Using `$(document).ready()` instead of `Drupal.behaviors`. Ready fires once; behaviors fire on every Ajax load.

2. **once() library** -- prevent duplicate processing
   - `once('uniqueId', '.selector', context)` returns elements not yet processed
   - jQuery pattern: `$(once('id', '.selector', context)).each(function() { ... })`
   - Vanilla JS: `once('id', document.querySelectorAll('.selector'), context)`
   - Declare `core/once` as library dependency
   - WRONG-WAY CALLOUT: Forgetting `once()` in behaviors. Without it, behaviors duplicate DOM changes on every Ajax load.

3. **drupalSettings** -- passing PHP values to JS
   - PHP: `$build['#attached']['drupalSettings']['module']['key'] = $value`
   - JS: `settings.module.key` in behavior attach function
   - Values must be JSON-serializable

4. **Ajax API basics**
   - Ajax form callbacks: return `AjaxResponse` with commands
   - Common commands: `ReplaceCommand`, `HtmlCommand`, `InvokeCommand`, `CssCommand`
   - Ajax form element: `'#ajax' => ['callback' => '::ajaxCallback', 'wrapper' => 'element-id']`
   - The callback method receives `$form` and `FormStateInterface $form_state`
   - Can return a render array (replaces wrapper content) or `AjaxResponse` with multiple commands

5. **States API** -- declarative form element visibility
   - `'#states' => ['visible' => [':input[name="field"]' => ['checked' => TRUE]]]`
   - States: visible, invisible, enabled, disabled, required, optional, checked, unchecked
   - Conditions: checked, unchecked, empty, filled, value, expanded, collapsed
   - No custom JS needed -- Drupal handles it
   - WRONG-WAY CALLOUT: Writing custom JS for show/hide form elements. Use #states API instead -- it's declarative and works with form rebuilds.
