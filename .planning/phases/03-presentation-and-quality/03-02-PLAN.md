- Cache tags: node:5, config:module.settings, node_list (list tag)

**SKILL.md body (under 500 lines):**

1. **The golden rule: EVERY render array needs #cache**
   - If it displays entity data -> add entity cache tags
   - If it displays config data -> add config cache tags
   - If it varies by user/role/URL -> add cache contexts
   - If it's always fresh -> consider lazy builders instead of max-age 0
   - WRONG-WAY CALLOUT: Omitting #cache on render arrays. Render arrays without cache metadata become stale after data changes. Caching is invisible during development but critical in production.

2. **Cache tags -- "What does this depend on?"**
   - Entity: `$entity->getCacheTags()` returns `['node:5']`
   - Entity list: `$entity_type->getListCacheTags()` returns `['node_list']` -- invalidated when ANY node is created/updated/deleted
   - Config: `$config->getCacheTags()` returns `['config:module.settings']`
   - Custom: `Cache::invalidateTags(['my_module:custom_tag'])` for custom invalidation
   - Tags are ADDITIVE -- use all that apply
   - Show complete example: render array with entity + config tags

3. **Cache contexts -- "What does this vary by?"**
   - `user`: unique per user (user-specific content)
   - `user.roles`: varies by role combination
   - `user.permissions`: varies by permission set (more granular than roles)
   - `url.path`: varies by current URL path
   - `url.query_args`: varies by query string
   - `languages`: varies by active language
   - Contexts BUBBLE UP to parent render arrays -- a child's context affects the whole page
   - Show example with user.roles context

4. **max-age -- "How long should this live?"**
   - `Cache::PERMANENT` (default) -- cached until tags are invalidated
   - `3600` -- cached for 1 hour (use for external data without invalidation tags)
   - `0` -- NEVER cache
   - WRONG-WAY CALLOUT: Setting max-age 0 without understanding consequences. max-age 0 bubbles up to the PAGE level, preventing the entire page from being cached by Dynamic Page Cache. Use lazy builders instead to isolate uncacheable content.
   - WRONG-WAY CALLOUT: Relying on max-age 0 for anonymous users. Internal Page Cache ignores bubbled max-age. For truly uncacheable anonymous content, use `\Drupal::service('page_cache_kill_switch')->trigger()`.

5. **Lazy builders -- isolate uncacheable content**
   - When a small part of the page is dynamic, use `#lazy_builder` instead of max-age 0
   - `$build['element'] = ['#lazy_builder' => ['service_name:method', [scalar_args]], '#create_placeholder' => TRUE]`
   - The lazy-built content is replaced at render time; the rest of the page remains cached
   - Arguments MUST be scalar (strings, numbers, booleans) -- no objects or arrays
   - Auto-placeholdering: Drupal automatically converts highly dynamic content to placeholders
   - WRONG-WAY CALLOUT: Passing non-scalar arguments to lazy builders. Lazy builder arguments must be JSON-serializable scalars. Pass entity IDs, not entity objects.
   - Show complete lazy builder example from book

6. **CacheableDependencyInterface**
   - Implement on custom value objects, access results, response objects
   - Methods: `getCacheTags()`, `getCacheContexts()`, `getCacheMaxAge()`
   - AccessResult already implements this -- access results carry cache metadata
   - `CacheableMetadata::createFromRenderArray()` and `::createFromObject()` for merging
   - `$metadata->applyTo($render_array)` to apply merged metadata

7. **Block plugin caching**
   - Override `getCacheContexts()` and `getCacheTags()` on block class
   - Return contexts/tags in addition to parent: `return Cache::mergeContexts(parent::getCacheContexts(), ['user.roles'])`
   - Default block caching is quite aggressive -- override when block content varies

8. **Cache API for custom storage**
   - `\Drupal::cache('bin_name')->get($cid)` / `->set($cid, $data, $expire, $tags)`
   - Default bin: `default`. Other bins: `render`, `page`, `discovery`, `data`
   - Tags on cache entries enable automatic invalidation via `Cache::invalidateTags()`
   - Use for expensive computations, external API responses, aggregated data

9. **Internal Page Cache vs Dynamic Page Cache**
   - Internal Page Cache (anonymous only): Caches full page response. Does NOT respect bubbled max-age. Use page_cache_kill_switch for uncacheable anonymous pages.
   - Dynamic Page Cache (authenticated users): Caches render arrays. Respects all cache metadata. Varies by contexts, invalidates by tags.
   - WRONG-WAY CALLOUT: Assuming max-age 0 prevents anonymous caching. Internal Page Cache ignores max-age. Anonymous users may see stale content. Use page_cache_kill_switch service.

10. **Cross-references**
    - "See also: drupal-theming (if installed) for render array structure and #theme patterns. Cache metadata goes on EVERY render array the theming skill teaches you to build. If not available, add `'#cache' => ['tags' => [], 'contexts' => [], 'max-age' => ...]` to all render arrays."
    - "See also: drupal-access-security (if installed) for AccessResult cache metadata. Access results carry cache tags and contexts that affect the render pipeline. If not available, always add `->addCacheContexts(['user.permissions'])` to AccessResult."
    - "See also: drupal-plugins-blocks (if installed) for block plugin caching overrides (getCacheContexts, getCacheTags). If not available, override these methods on BlockBase subclasses."
    - "See also: drupal-entities-fields (if installed) for entity cache tags via getCacheTags() and list cache tags."
