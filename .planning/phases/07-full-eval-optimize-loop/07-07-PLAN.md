     - What patterns does Sonnet already know well?

   - Comparison with Phase 6 iteration-1 data (for the 4 existing skills)

<task type="auto">
  <name>Task 2: Iterate on weak-delta skills and update analysis</name>
  <files>eval/analysis-full-13.md</files>
  <action>
For each skill classified as "zero delta" or "low delta" in Task 1:

1. **Diagnose why:** Read the grading.json evidence for both with/without runs.
   - Are assertions too generic? (Both pass easily)
   - Does Sonnet already know this pattern well? (Training data coverage)
   - Is the eval prompt not triggering the skill's wrong-way traps?

2. **Determine if iteration is feasible:**
   - If assertions are too generic: Tighten them to target more specific patterns. Update evals.json.
   - If Sonnet knows the pattern: Document this as a valid finding. The skill adds value for weaker models (as confirmed in Phase 6 with Opus vs Sonnet).
   - If the prompt doesn't trigger wrong-way traps: Revise the prompt to include trigger words. Update evals.json.

3. **If assertions or prompts were changed:**
   - Re-run the affected skill(s) through the eval pipeline (setup, run with/without, grade, teardown)
   - Update benchmark.json for affected skills
   - Update eval/analysis-full-13.md with iteration notes

4. **If no changes warranted:**
   - Document in analysis why the delta is low and that it's a valid empirical result
   - Note that skills may still add value for: weaker models, more ambiguous prompts, or edge cases

5. **Add final sections to analysis:**
   - "Iteration Log" -- what was tried, what changed, results
   - "Overall Verdict" -- does the skill set provide measurable value?
   - "Recommendations" -- which skills to prioritize, which might need future work
   - "E2E Verification Impact" -- what did agent-browser E2E assertions catch that code-only missed?

Do NOT spend more than 2 iteration cycles. If a skill still shows 0% delta after tightening, accept it as a valid result.
  </action>
