   - Root cause: Are assertions too generic? Is the pattern well-documented?

**Step 3: Identify iteration candidates**

For skills with <10% delta, determine:
- If assertions are too generic (e.g., "file exists" passes both ways) -> tighten
- If the skill teaches patterns already in training data -> accept low delta
- If the skill has wrong-way callouts not tested by assertions -> add assertions

List specific changes needed (assertion text changes, skill content additions) for each iteration candidate.

**Step 1: Apply iteration to identified weak-delta skills**

Based on Task 1 analysis, for each skill identified as needing iteration:

**Option A: Tighten assertions (if assertions are too generic)**
- Update the skill's evals.json with more discriminating assertions
- Re-run just that skill pair (setup, headless sonnet x2, teardown, grade)
- Update benchmark.json

**Option B: Accept low delta with documentation (if pattern is well-documented)**
- Add a note to the skill's section in analysis-full-13.md explaining why low delta is expected
- Example: "Routing controllers: Sonnet already knows ControllerBase/create() pattern from training data. Low delta is expected and acceptable."

**Option C: Improve skill content (if skill has untested differentiators)**
- Identify a specific wrong-way callout not covered by current assertions
- Add assertion targeting it
- Re-run and grade

**Iteration budget:** Maximum 2-3 skills can be re-run (each takes ~30 min). Prioritize skills where tightening assertions is likely to reveal a real delta (Option A), not skills where the model genuinely knows the pattern (Option B).

**Step 2: Update final analysis**
