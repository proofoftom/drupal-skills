## Critical Insight: Why 8/13 Skills Show 0% Delta

Sonnet-powered executors wrote evals.json files that test for what Sonnet already knows (standard Drupal patterns). The assertions are vacuous -- Sonnet passes them equally with and without the skill.

**Pattern: What makes an eval effective vs ineffective**

| Effective (High Delta) | Ineffective (0% Delta) |
|------------------------|----------------------|
| Tests anti-patterns Sonnet makes | Tests patterns Sonnet knows |
| "Does NOT use max-age: 0" | "Does use ControllerBase" |
| Specific format: `^10 \|\| ^11` | Generic: "has core_version_requirement" |
| Tests "why" knowledge | Tests "what" knowledge |
| Catches gotchas and edge cases | Checks happy path only |

## Execution Infrastructure Decisions

1. **No headless `claude -p` sessions**: They're black boxes -- hang silently, produce no output, can't intervene. Use regular Agent subagents instead for observability.
2. **1 executor agent per skill** (NON-NEGOTIABLE): Each handles exactly 1 skill, spawning 2 headless claude subagents (with + without). Multiple executors can run in parallel. Never batch multiple skills into one executor.
3. **Agent teams not needed for evals**: Teams add shared task lists and inter-agent messaging -- counterproductive for independent eval runs. Stick with subagents for isolation.
4. **GSD config balanced profile already uses Sonnet for executors**: No need for headless claude workaround.

## Differentiating Assertions Per Skill (From Research)

These are the SPECIFIC patterns each skill teaches that Sonnet would likely MISS without the skill. Evals must test for THESE, not standard patterns:

### drupal-routing-controllers
- No `\Drupal::service()` calls in controller classes (use DI)
- `_form` route key for forms, `_controller` for controllers (not interchangeable)
- Error responses (not exceptions) for invalid input
- Constructor params are private (not public)

### drupal-forms-api
- `setErrorByName()` for validation (not exceptions or messages)
- Config schema types: `label` (translatable) vs `string` (machine-readable)
- `parent::buildForm()` for auto submit button; `parent::submitForm()` for auto message
- ConfigFormBase MUST be paired with config/schema/*.schema.yml

### drupal-plugins-blocks
- 4-parameter `create()` signature (NOT 2-parameter like controllers)
- `parent::__construct($configuration, $plugin_id, $plugin_definition)` with 3 params
- `blockForm()`/`blockSubmit()` (not generic `submitForm()`)
- `defaultConfiguration()` returns array of config defaults
- D11: `#[Block(...)]` with `new TranslatableMarkup()`, NOT `@Block` with `@Translation()`

### drupal-config-storage
- Config vs State vs TempStore decision tree (Config = exportable settings, State = env-specific, TempStore = temporary)
- `getEditable()` for writes (not `\Drupal::config()`)
- Schema `label` type for translatable human-readable strings
- Config YAML needs `langcode` and `status` keys

### drupal-access-security
- `AccessResult` objects (not booleans) -- booleans break cache-aware access
- Cache contexts on AccessResult: `->addCacheContexts(['user.permissions'])`
- CSRF via route requirement `_csrf_token: 'TRUE'` (not manual token validation)
- Every permission string must be declared in `.permissions.yml` or silently denies

### drupal-theming
- Template naming: underscores in hook_theme() -> hyphens in filename (my_module_item -> my-module-item.html.twig)
- `hook_theme()` MUST declare ALL variables with defaults
- CSS/JS via `#attached['library']` (NEVER inline `<script>`/`<style>` tags)
- Return render arrays from controllers (NEVER plain strings)

### drupal-caching (ALREADY HIGH DELTA: +75%)
- Golden rule: EVERY render array needs `#cache` with tags, contexts, max-age
- `max-age: 0` kills ENTIRE page cache via bubbling (use `#lazy_builder` instead)
- Cache tags = data deps (`node:5`), contexts = request variations (`user.roles`)
- `page_cache_kill_switch` service needed for anonymous user cache control

### drupal-testing
- Use LOWEST sufficient test type (Unit < Kernel < Functional < FunctionalJavascript)
- `protected static $modules = [...]` required for Kernel tests
- `$this->installEntitySchema()` and `$this->installSchema()` not automatic
- `@group` annotation required for test discovery
- `$defaultTheme = 'stark'` required for BrowserTestBase

### drupal-database-api
- NEVER use Database API for entity data (use Entity Query)
- Placeholders (`:placeholder`) always, NEVER string concatenation (SQL injection)
- `addTag('my_tag')` enables `hook_query_alter()` support
- Update hooks must pair with hook_schema() changes for existing sites
- Transactions with try/finally for rollback safety

### drupal-views-dev
- Entity data uses `views_data` handler class in annotation (NEVER `hook_views_data()` for entities)
- `group` key required in Views data definitions (missing = "Broken handler" error)
- Virtual/computed ViewsField plugins MUST override `query()` with empty body
- `hook_views_data()` for NEW tables; `hook_views_data_alter()` for MODIFYING existing

### drupal-batch-queue-cron
- `hook_cron()` only discovers/queues (NEVER processes heavy work)
- Queue name MUST match QueueWorker plugin ID exactly
- `cron: {time: 30}` annotation required or items never process
- `RequeueException` (retryable) vs `SuspendQueueException` (systemic) vs generic (lost)
- Lock API with `release()` in finally block

### drupal-module-scaffold (ALREADY HIGH DELTA: +43%)
- `core_version_requirement: ^10 || ^11` (not `core: 8.x` or just `^10`)
- Dependencies: `drupal:node` format (not bare `node`)
- PSR-4: ALL classes in `src/` (only `.module` file at root)
- `declare(strict_types=1)` in PHP files
- Only create `.module` file if hooks are needed

### drupal-entities-fields
- Config entities MUST have `config/schema/*.schema.yml` (most commonly forgotten file)
- `config_export` list required for config entities
- `setDisplayOptions()` mandatory for base fields to appear in forms/views
- `parent::baseFieldDefinitions()` must be called and merged
- Entity route providers auto-generate CRUD routes (don't hand-write)
