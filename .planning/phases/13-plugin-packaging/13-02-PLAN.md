FIRST: Test `--plugin-dir` + `-p` compatibility by running:
```bash
claude --plugin-dir . -p "What Drupal skills are available? List them all."
```

If this works (outputs skill names), the script can use headless mode. If it fails or produces no skill output, note the finding and design the script for interactive verification instead.

The test script should:

1. Define 12 test prompts (from the research) as an array, each with:
   - The natural Drupal prompt text
   - The expected skill name(s) that should activate

2. For each prompt, run `claude --plugin-dir /path/to/drupal-skills -p "<prompt>"` (if headless works) or provide instructions for interactive testing.

3. Check the output/behavior for evidence of skill activation. Possible signals:
   - The response uses patterns specific to the skill (e.g., caching skill patterns for caching prompts)
   - The response references SKILL.md content that baseline Claude wouldn't know
   - Use `--verbose` or `--debug` flags if available to see skill loading

4. Track results: pass/fail per prompt, total activation rate.

5. Output a summary: `X/12 prompts triggered correct skill(s) = Y% activation rate`

6. Exit 0 if rate >= 80% (10/12), exit 1 otherwise.

Important considerations:
- The script must work from the repo root directory
- Use absolute path for --plugin-dir (resolve from script location)
- Each prompt run is independent (no session state)
- If `--plugin-dir` + `-p` is incompatible, document this finding clearly and adapt: the script should print manual testing instructions instead of running headless, and the checkpoint task handles interactive verification
- Set a timeout per prompt (30s) to prevent hangs
