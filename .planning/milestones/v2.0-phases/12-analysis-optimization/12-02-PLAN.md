**New prompt:** Create a module (e.g., `team_profiles`) that renders team member profiles using the theme layer. Each profile should be rendered via a custom theme hook. The module must:
- Define a theme hook with variables for name, role, and bio
- Provide a default preprocess function that adds CSS classes based on the role (e.g., "team-member--lead" for leads)
- Implement `hook_theme_suggestions_HOOK()` to suggest alternative templates based on the role (e.g., `team_profile__lead` for the "lead" role)
- Provide a page route displaying sample profiles
- Attach a CSS library

**New expectations:**
1. `template_preprocess_HOOKNAME()` function exists in .module file with correct naming convention (the preprocess function MUST use `template_preprocess_` prefix followed by the hook name, NOT `MODULE_preprocess_HOOKNAME` for the default preprocessor -- MODULE_preprocess is for module-specific overrides, template_preprocess is the default)
2. Preprocess function modifies `$variables` by reference and adds/modifies attributes or classes (NOT just passing through variables unchanged)
3. `hook_theme_suggestions_HOOK()` or `hook_theme_suggestions_HOOK_alter()` implemented to provide dynamic template suggestions based on a variable value (e.g., role)
4. Template suggestions use double-underscore pattern: `base_hook__context` (NOT hyphen or single underscore -- Drupal maps double underscores to directory separators for template discovery)
5. hook_theme() declares ALL template variables with default values in 'variables' key
6. Template filename uses hyphens (not underscores)
7. Module enables: ddev drush en returns exit 0
8. Browser: verify themed output renders on the page route (via eval-browser)
9. phpcs passes

**Key differentiators:**
- `template_preprocess_HOOKNAME` naming is specific -- SKILL.md line 187 teaches it but Haiku commonly uses `MODULE_preprocess_HOOKNAME` instead
- `hook_theme_suggestions_HOOK()` is obscure -- Haiku rarely implements it correctly
- Double-underscore pattern for suggestions is a Drupal convention Haiku frequently gets wrong

**Entities-fields eval -- target bundle entity pattern:**

Replace the current entities-fields eval with a harder one requiring bundle entity wiring.

**New prompt:** Create a module (e.g., `project_tracker`) that provides a content entity type `ProjectItem` with bundle support. Different project types (task, milestone, deliverable) should be separate bundles. The module must:
- Define a content entity type `ProjectItem` with base fields: title (string), status (list_string), deadline (timestamp)
- Define a companion config entity type `ProjectItemType` that acts as the bundle entity
- Wire the bundle relationship via annotations/attributes
- Include all necessary handlers (forms, list builder, access, route provider)
- Provide admin routes for managing both the entity and its types

**New expectations:**
1. Content entity annotation/attribute includes `bundle_entity_type` key pointing to the config entity type ID (without this, Drupal doesn't know which config entity provides bundles)
2. Config entity annotation/attribute includes `bundle_of` key pointing to the content entity type ID (this creates the reverse relationship -- without it, the config entity isn't recognized as a bundle provider)
3. Content entity has `bundle` key in `entity_keys` annotation (without this, Drupal cannot determine which bundle an entity instance belongs to)
4. Config entity class exists separately from content entity class -- one per file, config entity extends ConfigEntityBase or ConfigEntityBundleBase (NOT mixing content and config entity in one class)
5. Content entity uses `EntityChangedTrait` AND has a `changed` base field definition (the trait provides getChangedTime/setChangedTime but requires the base field to exist -- having the trait without the field or vice versa is an incomplete pattern)
6. Content entity baseFieldDefinitions() calls parent::baseFieldDefinitions($entity_type) (MUST call parent or core base fields are lost)
7. Module enables: ddev drush en returns exit 0
8. Browser: navigate to entity type admin page, verify bundle management UI loads (via eval-browser)
9. phpcs passes

**Key differentiators:**
- `bundle_entity_type` / `bundle_of` wiring is complex and frequently wrong
- EntityChangedTrait + changed field is a pair that must both be present
- Config entity as bundle provider requires specific annotation keys Haiku often omits
- SKILL.md lines 19-31 and 50-145 teach these patterns explicitly
