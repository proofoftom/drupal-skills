**Step 2: eval-grader validation (INFRA-02)**

- ddev project: `d10-smoke-test`
- Output path: `/tmp/d10-smoke-test/grading.json`
- After agent completes: validate the output matches grading.json schema using jq:
  - Has `expectations` array where each item has `text`, `passed`, `evidence`
  - Has `summary` object with `passed`, `failed`, `total`, `pass_rate`

If eval-grader fails, fix `.claude/agents/eval-grader.md` and retry.

**Step 3: Empirical test of skills: frontmatter for knowledge isolation (Research Open Question #1)**
The research doc notes uncertainty about whether `skills:` frontmatter can reference skills in `skills/` (non-standard path vs `.claude/skills/`). Test this empirically:

Option A -- Try `skills:` frontmatter:
1. Create a temporary copy of eval-executor.md with `skills: [drupal-caching]` added to frontmatter
2. Spawn it with a simple task ("What Drupal caching patterns do you know about?")
3. Check if the response references SKILL.md content (cache golden rule, render arrays, etc.)

Option B -- If `skills:` does not resolve project skills:
1. Try symlinking: `ln -s ../../skills/drupal-caching .claude/skills/drupal-caching` and retry
2. If symlink works, document that `.claude/skills/` symlinks are needed
3. If neither works, confirm Read-based loading is the correct approach

Document the result in the SUMMARY: which mechanism works, which was chosen, and why.

If `skills:` frontmatter works (with or without symlinks) AND provides better isolation than Read-based (skill content injected before first turn, impossible to forget), update `.claude/agents/eval-executor.md` to note the recommended approach in a comment. Do NOT add `skills:` permanently -- the orchestrator controls this per-run.

Clean up any symlinks created during Step 3:
