**SKILL.md body (MUST be under 500 lines -- this is the highest risk for exceeding the limit):**

Use aggressive progressive disclosure: main SKILL.md covers decisions and core patterns ONLY. Detailed field type lists, handler option catalogs, and file/image specifics go in references/.

1. **Decision tree: "What kind of entity do you need?"**
   - Stores content (user-created data with fields, revisions, translations) -> ContentEntityType
   - Stores configuration (admin-defined settings, exported to YAML) -> ConfigEntityType
   - "Do you need revisions?" -> add revision keys + RevisionLogInterface
   - "Do you need bundles?" -> add bundle entity type + bundle_of
   - "Does it need admin UI?" -> add handlers (list_builder, form, route_provider) + links

2. **Content entity type definition -- D10/D11 DUAL SYNTAX SIDE-BY-SIDE**
   - Show COMPLETE ContentEntityType with all common keys: id, label, handlers, base_table, entity_keys, links
   - D10 annotation format (full example with @ContentEntityType)
   - D11.1+ attribute format (full example with #[ContentEntityType])
   - Key differences highlighted: = vs :, @Translation vs new TranslatableMarkup, { } vs [ ], string vs ::class
   - WRONG-WAY CALLOUT: Mixing annotation and attribute syntax (e.g., using = inside #[...])
   - WRONG-WAY CALLOUT: Using @Translation inside attribute syntax

3. **Config entity type definition -- D10/D11 DUAL SYNTAX**
   - Show ConfigEntityType with: id, label, handlers, config_prefix, entity_keys, links, config_export
   - D10 annotation and D11 attribute side-by-side
   - config/schema/*.schema.yml MUST be paired with config entity
   - WRONG-WAY CALLOUT: ConfigEntityType without schema file -- schema is REQUIRED for config export, translation, and validation
   - WRONG-WAY CALLOUT: Missing config_export list -- without it, config cannot be exported

4. **Entity handlers -- decision tree**
   - "Do you need a custom list page?" -> NO: use EntityListBuilder default. YES: extend EntityListBuilder
   - "Do you need custom forms?" -> NO: use ContentEntityForm. YES: extend ContentEntityForm
   - "Do you need custom routes?" -> NO: use AdminHtmlRouteProvider. YES: extend DefaultHtmlRouteProvider
   - "Do you need custom access?" -> NO: use EntityAccessControlHandler. YES: extend it
   - WRONG-WAY CALLOUT: Hand-writing entity CRUD routes when AdminHtmlRouteProvider generates them automatically
   - WRONG-WAY CALLOUT: Hand-writing entity forms when ContentEntityForm auto-builds from base field definitions

5. **Base field definitions (COMPACT)**
   - Pattern: `baseFieldDefinitions()` method, parent call, BaseFieldDefinition::create()
   - Common field types (brief list): string, text_long, integer, boolean, entity_reference, created, changed, email, uri
   - Display options: setDisplayOptions('view', ...), setDisplayOptions('form', ...), setDisplayConfigurable
   - Show ONE complete example with 3-4 fields, not exhaustive catalog
   - "For file and image fields, see references/files-images.md"

6. **Entity interface and complete file ecosystem**
   - Interface pattern: ProductInterface extends ContentEntityInterface
   - Show complete ecosystem for a content entity: Entity class, Interface, ListBuilder, Form, .module (entity_type_build if needed), .routing.yml (if not using route_provider), .permissions.yml, config/schema/*.schema.yml (for config entities)
   - Directory tree showing all files

   - "See also: drupal-routing-controllers (if installed) for custom routes beyond entity route providers."
