Review and optimize the `description:` field in YAML frontmatter of all 13 SKILL.md files as a holistic system. The descriptions compete with each other for activation -- changing one affects all others.

**Read all 13 current descriptions first.** Then apply these optimization principles from research:

1. **Partition the Drupal space cleanly:** Each description should own a distinct area. Address known overlap risks:
   - "render arrays": caching = cache metadata ON render arrays; theming = building/templating render arrays. Make this distinction explicit.
   - "services/DI": routing = defining services and injecting into controllers; plugins-blocks = injecting into plugins via create(). Distinguish.
   - "entity": entities-fields = creating entity types; database-api already has "Do NOT use for entity data" (good, keep it).

2. **Include user-facing phrases:** Start with what the skill does, then "Use when asked to..." using vocabulary users actually use (not just API names). Include synonyms where helpful (e.g., "settings page" alongside "configuration form").

3. **Add negative triggers where disambiguation is needed:** Use "Do NOT use for..." to prevent over-triggering on ambiguous terms. Key disambiguations:
   - drupal-caching: "Do NOT use for building templates or themed output" (that's theming)
   - drupal-theming: "Do NOT use for cache invalidation or cache metadata strategy" (that's caching)
   - drupal-routing-controllers: "Do NOT use for form routes" (that's forms)
   - drupal-forms-api: include "settings page" to capture that common phrasing
   - drupal-database-api: keep existing "Do NOT use for entity data"

4. **Be slightly pushy:** Per official Claude docs, skills tend to under-trigger. Descriptions should be a little aggressive about when they apply. Use "Use WHENEVER..." for core skills that apply broadly (scaffold, caching, theming).

5. **Keep under 1024 characters each, aim for 2-4 lines.**

6. **Preserve all content below the frontmatter `---` delimiter.** Only modify the `description:` value in the YAML frontmatter block.

After updating descriptions, verify no two skills have descriptions that would both fully match on the same narrow prompt (some overlap on multi-skill prompts is expected and desired).

IMPORTANT: When editing SKILL.md files, ONLY change the description field in the YAML frontmatter. Do NOT modify any content below the closing `---` of the frontmatter block. Some SKILL.md files have a stray `description:` line in the body content (drupal-module-scaffold and drupal-routing-controllers have example .info.yml snippets with `description:` keys) -- do NOT touch those.
