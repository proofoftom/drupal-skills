<task type="auto">
  <name>Task 1: Verify skills directory and create install.sh</name>
  <files>install.sh</files>
  <action>
First, verify PACK-01: confirm `skills/` contains all 13 drupal-* directories, each with a SKILL.md file. List them and verify count equals 13.

Then create `install.sh` following the pattern from research. The script must:

1. Use `#!/usr/bin/env bash` shebang and `set -euo pipefail`
2. Resolve SCRIPT_DIR to the directory containing install.sh
3. Set SKILLS_SOURCE="$SCRIPT_DIR/skills" and SKILLS_TARGET="${HOME}/.claude/skills"
4. Validate that skills/ directory exists
5. `mkdir -p "$SKILLS_TARGET"`
6. Parse arguments: `--symlink` flag switches from copy to symlink mode
7. Loop over each `drupal-*/` directory in SKILLS_SOURCE:
   - Print "Installing: skill-name" or "Updating: skill-name" if target exists
   - If target exists (file or symlink), remove it first with `rm -rf`
   - Copy (`cp -r`) or symlink (`ln -s`) based on mode
   - Increment counter
8. Print summary: "Installed N Drupal skills to $SKILLS_TARGET"
9. Print: "Skills will be available in your next Claude Code session."

Make the script executable: `chmod +x install.sh`

Handle edge cases per research pitfalls:
- Always mkdir -p before copying (Pitfall 3)
- Default to copy, --symlink for power users (Pitfall 4)
- Check for existing skills and handle gracefully (update vs install messaging)
- Use absolute path for symlinks (resolve with readlink -f or cd+pwd)
  </action>
  <verify>
    <automated>ls skills/drupal-* -d | wc -l | xargs test 13 -eq && test -x install.sh && bash -n install.sh && echo "PASS: 13 skills found, install.sh is executable and syntactically valid"</automated>
  </verify>
  <done>skills/ contains all 13 drupal-* directories (PACK-01 verified). install.sh is executable, syntactically valid, supports --symlink flag, and handles existing installations gracefully.</done>
</task>

Create README.md following the structure from research, written from the USER's perspective (not the builder's perspective -- per Pitfall 5).

Structure:

1. **Title and tagline:** "# Drupal Skills for Claude" followed by a one-line description: "13 Claude Code skills for generating correct, production-ready Drupal 10/11 module code."

2. **Quick Start** section (3 lines: clone, cd, install):
   ```
   git clone <repo-url>
   cd drupal-skills
   ./install.sh
   ```

3. **What's Included** section: Table of all 13 skills with columns: Skill Name, What It Does. Use the skill descriptions as basis but write short human-readable summaries (not the full description field). Order by dependency wave:

   Wave 1 (Foundations):
   - drupal-module-scaffold: Scaffold modules with .info.yml, PSR-4, .module
   - drupal-routing-controllers: Routes, controllers, services, dependency injection
   - drupal-entities-fields: Content and config entity types, base fields, handlers

   Wave 2 (Core Workflow):
   - drupal-forms-api: Form API lifecycle, form altering, settings forms
   - drupal-plugins-blocks: Block plugins, custom plugin types, plugin DI
   - drupal-config-storage: Config API, State API, TempStore, config schemas
   - drupal-access-security: Permissions, access control, CSRF/XSS prevention

   Wave 3 (Presentation and Quality):
   - drupal-theming: Render arrays, Twig templates, theme hooks, libraries
   - drupal-caching: Cache tags, contexts, max-age, lazy builders
   - drupal-testing: PHPUnit tests (unit, kernel, functional, browser)
   - drupal-database-api: Schema API, dynamic queries, update hooks

   Wave 4 (Specialized):
   - drupal-views-dev: Views data integration, custom Views plugins
   - drupal-batch-queue-cron: Batch API, queue workers, cron hooks

4. **Usage** section: Explain that skills activate automatically based on prompt content. Give 3-4 example prompts showing different skills in action:

   - "Create a custom content entity type for events" (triggers entities)

7. **Drupal Version Support** section: Brief note that skills cover Drupal 10 with D11 PHP attribute syntax shown alongside D10 annotations.

8. **License** section: Add appropriate license notice (MIT or similar, check if repo has LICENSE file first).
