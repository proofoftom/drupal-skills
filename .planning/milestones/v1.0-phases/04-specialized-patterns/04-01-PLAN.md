Key Views API patterns from research:
- Entity Views: "views_data" handler in entity annotation -> EntityViewsData or custom subclass
- hook_views_data(): procedural hook returning table/field definitions
- Table definition: $data['table']['table']['group'], $data['table']['table']['base']
- Field responsibilities: 'field' => ['id' => 'plugin_id'], 'filter' => [...], 'sort' => [...], 'argument' => [...], 'relationship' => [...]
- Built-in field plugins: numeric, standard, serialized, custom, date, boolean, machine_name
- Built-in filter plugins: string, numeric, boolean, date, in_operator, bundle
- Custom ViewsField: extends FieldPluginBase, override render(ResultRow $values)
- Virtual fields: override query() with empty body to prevent SQL column error
- Custom ViewsFilter: extends InOperator for select list filters
- D10 annotations: @ViewsField("id"), @ViewsFilter("id"), @ViewsArgument("id")
- D11 attributes: PHP attribute equivalents (follow pattern from drupal-plugins-blocks)
- Plugin config schema: views.field.PLUGIN_ID in module.schema.yml

**SKILL.md body (under 500 lines):**

1. **Decision tree: "How do you expose data to Views?"**
   - Content entity type? -> Add "views_data" handler to entity annotation/attribute
     - Default fields sufficient? -> Use `Drupal\views\EntityViewsData` directly
     - Need custom fields or overrides? -> Extend EntityViewsData, override getViewsData()
   - Custom database table (non-entity)? -> Implement hook_views_data()
     - Define table as base table (standalone Views listing)
     - Define fields with plugin responsibilities: field, filter, sort, argument, relationship
   - Modifying existing Views data? -> Implement hook_views_data_alter()
     - Add fields to existing tables, change plugin IDs
   - WRONG-WAY CALLOUT: Writing hook_views_data() for content entities. Use the EntityViewsData handler instead -- it auto-generates Views integration for all base fields. Only use hook_views_data() for custom (non-entity) database tables.

2. **Entity Views integration (one line)**
   - In entity annotation handlers: `"views_data" = "Drupal\views\EntityViewsData"`
   - Or custom: `"views_data" = "Drupal\my_module\Entity\MyEntityViewsData"`
   - Custom class extends EntityViewsData, overrides getViewsData()
   - Show entity annotation with views_data handler

3. **hook_views_data() -- exposing a custom table**
   - Function signature: `function my_module_views_data() { return $data; }`
   - Table group: `$data['table_name']['table']['group'] = t('Label')` -- ALWAYS set for UI organization
   - Base table: `$data['table_name']['table']['base']` with field (primary key), title, help
   - WRONG-WAY CALLOUT: Omitting the table group in hook_views_data(). Without `$data['table']['table']['group']`, your fields scatter across the Views UI among hundreds of other fields with no logical grouping.
   - Show complete hook_views_data() example from book (sports/players table with group, base table, fields)

4. **Field responsibilities in hook_views_data()**
   - Each column entry can have multiple responsibilities: field, filter, sort, argument, relationship
   - 'field' => ['id' => 'plugin_id'] -- how to display the value
   - 'filter' => ['id' => 'plugin_id'] -- how to filter by this column
   - 'sort' => ['id' => 'plugin_id'] -- how to sort by this column
   - 'argument' => ['id' => 'plugin_id'] -- how to use as contextual filter
   - 'relationship' => ['base' => 'other_table', 'base field' => 'id', 'id' => 'standard'] -- JOINs

   Common built-in field plugins: `numeric`, `standard`, `date`, `boolean`, `serialized`
   Common built-in filter plugins: `string`, `numeric`, `boolean`, `date`, `in_operator`
   Common built-in sort plugins: `standard`, `date`

   Show complete field definition with field + filter + sort on same column

5. **Relationships (JOINs between tables)**
   - Define in hook_views_data() via 'relationship' key
   - base: target table, base field: join column, id: 'standard', label: UI label
   - Show relationship example (player -> team)

6. **Custom ViewsField plugin -- when built-in plugins are not enough**
   - When: computed data, cross-entity lookups, custom rendering logic
   - Plugin namespace: `Drupal\my_module\Plugin\views\field`
   - D10: `@ViewsField("plugin_id")` annotation
   - D11: PHP attribute equivalent (follow pattern from other plugin types)
   - Extends FieldPluginBase
   - Override render(ResultRow $values) for custom output
   - WRONG-WAY CALLOUT: Forgetting to override query() for virtual fields. If your field has no database column (computed/virtual data), you MUST override query() with an empty method body. Otherwise Views adds a non-existent column to SQL, causing a database error.
   - Show complete custom ViewsField example (book's ProductImporter)

7. **Custom ViewsFilter plugin -- custom filter options**
   - Common pattern: extend InOperator for select-list filters
   - Plugin namespace: `Drupal\my_module\Plugin\views\filter`
   - D10: `@ViewsFilter("plugin_id")` annotation
   - D11: PHP attribute equivalent
   - Override init() to set valueTitle and options callback
   - Show complete custom ViewsFilter example (book's TeamFilter)

8. **Views plugin configuration schema**
   - Custom options in defineOptions()/buildOptionsForm() need schema
   - Define in module.schema.yml using dynamic types: `views.field.PLUGIN_ID`, `views.filter_value.PLUGIN_ID`
   - WRONG-WAY CALLOUT: Adding custom plugin options without configuration schema. Views plugins are stored as part of View config entities. Missing schema causes config export/import failures and validation errors.

9. **hook_views_data_alter() -- modifying existing data**
   - Add new virtual fields to existing tables (e.g., node_field_data)
   - Change plugin IDs for existing fields
   - Useful for adding cross-module fields to core entity tables

10. **Cross-references**
    - "See also: drupal-entities-fields (if installed) for entity type definitions and handlers where views_data handler is declared. If not available, add `'views_data' => 'Drupal\views\EntityViewsData'` to entity annotation handlers array."
    - "See also: drupal-plugins-blocks (if installed) for plugin discovery patterns, D10 annotations vs D11 attributes, and ContainerFactoryPluginInterface for DI in plugins. If not available, inject services via ContainerFactoryPluginInterface::create()."
    - "See also: drupal-database-api (if installed) for hook_schema() to define the custom tables that hook_views_data() exposes. If not available, define tables in module.install hook_schema()."
