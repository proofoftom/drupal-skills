**SKILL.md body (under 500 lines):**

1. **Decision tree: "How should you process data in the background?"**
   - Processing triggered by user action (form submit)?
     - Data fits in one request? -> Process directly in submit handler
     - Too much data? -> Use Batch API (BatchBuilder)
       - From form submit? -> batch_set() (Form API triggers automatically)
       - From Drush/code? -> batch_set() + drush_backend_batch_process()
   - Processing should happen periodically?
     - Simple periodic task (bounded work)? -> hook_cron()
     - Variable-size workload? -> Add items to queue in hook_cron, use QueueWorker
   - Processing queue items on demand?
     - Via Drush? -> `drush queue:run queue_name`
     - Via custom code? -> Programmatic claim/process/delete loop
   - WRONG-WAY CALLOUT: Unbounded processing in hook_cron(). For variable-size workloads, use QueueWorker with `cron = {"time" = N}` to process items with a time budget. Unbounded loops block other cron tasks and can time out.

2. **Batch API -- multi-request processing for large operations**
   - Use BatchBuilder (not manual batch definition arrays)
   - Setup: `(new BatchBuilder())->setTitle()->setFinishCallback()->addOperation()`
   - Start: `batch_set($batch_builder->toArray())`
   - Drush context: check `PHP_SAPI == 'cli'` then `drush_backend_batch_process()`
   - Show complete BatchBuilder setup example from book

3. **Batch operations -- the processing function**
   - Signature: `function operationName($args, &$context)`
   - $context keys (CRITICAL -- do not confuse):
     - `$context['sandbox']` -- progress tracking within this operation (reset per operation call)
     - `$context['results']` -- accumulates data across ALL operations (shared, persists)
     - `$context['finished']` -- float 0-1 signaling completion (set to progress/max)
     - `$context['message']` -- real-time status text shown to user
   - Multi-request pattern: initialize sandbox on first call, splice items, update finished
   - WRONG-WAY CALLOUT: Mixing up batch $context keys. Using `$context['results']` for progress tracking (wrong -- use `$context['sandbox']`). Using `$context['sandbox']` to accumulate outcomes (wrong -- use `$context['results']`). Forgetting `$context['finished']` (batch never completes).
   - Show complete batch operation with multi-request processing from book

4. **Batch finished callback**
   - Signature: `function finished($success, $results, $operations)`
   - $success: boolean (all operations completed without fatal error)
   - $results: accumulated from $context['results'] across operations
   - $operations: any unfinished operations (if $success is FALSE)
   - Show complete finished callback example from book

5. **hook_cron -- periodic tasks**
   - Implement in .module file: `function my_module_cron() { ... }`
   - Keep processing bounded -- only do fixed-amount work
   - Use for: cleanup tasks, aggregation, queue population
   - Do NOT use for: processing unknown-size data sets (use QueueWorker)
   - Show hook_cron example from book

6. **QueueWorker plugins -- cron-based queue processing**
   - Plugin namespace: `Drupal\my_module\Plugin\QueueWorker`
   - D10 annotation: `@QueueWorker(id = "queue_name", title = @Translation("Label"), cron = {"time" = 10})`
   - D11: PHP attribute equivalent
   - Extends QueueWorkerBase, implements ContainerFactoryPluginInterface for DI
   - Override processItem($data) -- receives one item at a time
   - WRONG-WAY CALLOUT: QueueWorker plugin ID not matching queue name. For cron-based processing, the @QueueWorker id MUST exactly match the queue name used in `\Drupal::queue('name')->createItem()`. Mismatched names mean items are never processed.
   - Show complete QueueWorker plugin from book (TeamCleaner)

7. **Creating queue items**
   - `$queue = \Drupal::queue('queue_name')` or inject QueueFactory
   - `$queue->createItem($data)` -- $data can be any serializable value (stdClass, array)
   - Typically populate queue in hook_cron or form submit, process via QueueWorker
   - Show queue item creation example

8. **Programmatic queue processing**
   - Claim/process/delete loop for on-demand processing
   - Handle SuspendQueueException (break loop -- queue has systemic problem)
   - Handle generic Exception (log and continue -- skip bad item)
   - Show programmatic queue processing example from book

9. **Lock API -- preventing parallel execution**
   - Inject `lock` service (`Drupal\Core\Lock\LockBackendInterface`)
   - `$lock->acquire('lock_name')` returns TRUE if acquired, FALSE if already locked
   - Always `$lock->release('lock_name')` when done
   - WRONG-WAY CALLOUT: Forgetting to release locks. Always call `$lock->release()` after processing completes. Unreleased locks prevent future executions until the lock lease expires.
   - Show Lock API usage example from book

10. **Cross-references**
    - "See also: drupal-forms-api (if installed) for form submit handlers where batch_set() is called. If not available, call batch_set() in your form's submitForm() method."
    - "See also: drupal-plugins-blocks (if installed) for plugin discovery patterns, ContainerFactoryPluginInterface for DI in QueueWorker plugins. If not available, implement ContainerFactoryPluginInterface::create() to inject services."
    - "See also: drupal-database-api (if installed) for database operations commonly performed in batch/queue/cron processing. If not available, use \Drupal::database() for database access."

**references/logging-mail-tokens.md** (no frontmatter, just content):

Title: "Logging, Mail, and Tokens"

Cover three related APIs from book Ch 3:

**1. Logging (PSR-3 Logger Channels)**
- Getting a logger: `\Drupal::logger('my_module')` or inject `logger.factory` service
- Log levels: emergency, alert, critical, error, warning, notice, info, debug
- Usage: `$this->logger->error('Import failed for @item', ['@item' => $name])`
- Custom logger channel in services.yml: `logger.channel.my_module` with `parent: container.trait.logger_channel_factory` tag
- Placeholder syntax: @variable (sanitized), %variable (emphasized), :variable (URL)
- WRONG: Using watchdog() -- deprecated since D8. Use logger channels.

**2. Mail (hook_mail and Mail Manager)**
- Define message templates: hook_mail($key, &$message, $params)
- Set $message['subject'] and $message['body'][] (body is array of strings)
- Send mail: `\Drupal::service('plugin.manager.mail')->mail('my_module', 'key', $to, $langcode, $params)`
- Alter other modules' mail: hook_mail_alter(&$message)
- WRONG: Using PHP mail() directly. Use Mail Manager for pluggable backends, proper formatting, and hook_mail_alter support.

**3. Token API**
- Define custom tokens: hook_token_info() returns 'types' and 'tokens' arrays
- Implement token replacement: hook_tokens($type, $tokens, $data, $options, $bubbleable_metadata)
- Use Token service to replace: `\Drupal::token()->replace('[my_module:token_name]', $data)`
- Standard token types: node, user, site, date, current-user
- WRONG: Building custom placeholder systems. Use Token API for standardized, discoverable, UI-friendly replacements.

Keep reference file focused and practical. Skip custom mail plugin creation and custom logger implementation (niche use cases per research recommendation).
